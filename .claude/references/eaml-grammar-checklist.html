01⚙️ Foundational Grammar Design Decisions 0/0▼
Grammar Formalism
  • Choose and document your EBNF notation variant. W3C EBNF, ISO 14977, or ANTLR notation. Pick one and use it consistently. Metacharacters like ::=, |, { }, [ ], ( ) must be unambiguously defined.
    Recommended: W3C-style. Document the chosen notation at the top of grammar.ebnf with a legend of all metacharacters used.
    Critical
  • Define exactly one root production rule. The grammar must have a single start symbol — name it Program or SourceFile. Everything in a valid EAML file is derivable from this single rule.
    Program ::= Declaration* EOF — this one rule determines what constitutes a valid EAML file. EOF must be a defined terminal.
    Critical
  • Decide and document LL(k) target. Hand-written recursive descent parsers are LL(1) or LL(2). If any rule requires more than 2 tokens of lookahead, redesign that rule. Document the maximum k at each conflict point.
    LL(3+) lookahead is a grammar design smell. If you find one, it means a rule is ambiguous and needs to be restructured — not that the parser should look ahead further.
    Critical
  • Define terminal vs. non-terminal naming conventions. Terminals in ALL_CAPS or 'quoted'. Non-terminals in PascalCase. Be consistent throughout — a violation is a bug hiding as a typo.
    High
  • Define all whitespace handling rules explicitly. Whitespace is insignificant between tokens EXCEPT inside template strings and Python blocks. Define a WS terminal and state explicitly that it is skipped by the lexer, with the two exceptions named.
    WS ::= (' ' | '\t' | '\n' | '\r')+ — SKIPPED. Exceptions: (1) inside PYTHON_BLOCK captures, (2) inside template string text segments. State this in the grammar header.
    Critical
  • Define comment syntax and whether comments are preserved in the AST. Single-line //? Block /* */? Doc comments ///? Are comments attached to declarations for tooling use?
    Recommendation: // single-line only for MVP. Comments are skipped by the lexer (not in the AST). Doc comments (///) are Post-MVP. Reserve the /// syntax now so it doesn't conflict later.
    High
  • Define character encoding and identifier character sets. UTF-8 file encoding. Identifier characters: ASCII letters, digits, underscore only (recommended for MVP). String literals: any UTF-8 content.
    ASCII-only identifiers prevent subtle normalization bugs and keep the lexer simple. Unicode string content is needed for multi-language prompt text — allow it in string literals.
    High
  • Confirm full case sensitivity. Keywords are lowercase. Type names are PascalCase. These are conventions, but confirm that the grammar is case-sensitive (it should be) and document it.
    High
Structural Decisions
  • Define declaration ordering rules — are forward references permitted? Can a prompt reference a schema declared later in the file? Recommend: yes, via two-pass name resolution in semantic analysis.
    Add this as a grammar note: "Declarations may appear in any order within a Program. Name resolution is performed in a separate pass after parsing." This frees users from C-style ordering constraints.
    Critical
  • Define whether duplicate declaration names are a grammar error or semantic error. schema Foo appearing twice — the grammar allows it, semantic analysis rejects it. Document this boundary explicitly.
    High
  • Define the import declaration grammar for MVP even if single-file only. Reserve the syntax so multi-file support doesn't break existing files. ImportDecl ::= 'import' ModulePath ('{' NameList '}')? ';'?
    Medium
  • Tag every grammar construct as MVP or Post-MVP in the grammar document. Use a comment convention: // MVP or // Post-MVP. This prevents scope creep and makes it clear what the parser needs to handle at first release.
    Critical
02🔤 Lexical Layer — Tokens & Terminals 0/0▼
Keywords
  • Enumerate every reserved keyword and verify none collide with common engineering identifier names. Every keyword is permanently reserved — a user cannot name a schema after it.
    EAML keyword candidates: model, schema, prompt, tool, agent, pipeline, requires, import, let, await, if, else, return, true, false, null, python. Risk: model is extremely common in engineering vocabulary. Consider whether it can be both a keyword and a valid identifier in non-declaration contexts.
    Critical
  • Define keyword recognition via maximal munch — identifier first, keyword lookup second. The lexer scans the longest possible identifier token, then checks if it is a keyword. model_id is IDENT, not KW_MODEL + something.
    Critical
  • Define contextual keywords — words valid as identifiers outside their specific grammar positions. system, user, temperature, max_tokens, caps, description appear as field names inside declaration bodies but should be valid schema field names too.
    Recommendation: these are contextual — the parser gives them special meaning in specific positions, but the lexer always emits them as IDENT. This means schema Foo { system: String } is valid even though system is a prompt body field name.
    Critical
Identifiers & Literals
  • Define the identifier grammar precisely. Start character, continuation characters, length limit (or unlimited), leading underscore validity.
    IDENT ::= [a-zA-Z_][a-zA-Z0-9_]* (minus all keywords). Verify: does __init__ need to be valid? If tools can have names that match Python conventions, yes.
    Critical
  • Define string literal grammar including all supported escape sequences. Which are valid: \n, \t, \\, \", \uXXXX? What is the behavior on unknown escape like \q?
    STRING ::= '"' (ESC | ~["\\\n])* '"' where ESC ::= '\\' [ntr\\"'\{\}]. Note: include \{ and \} to allow literal braces in template strings. Unknown escapes = parse error (not silent pass-through).
    Critical
  • Define integer and float literal grammar. Decimal only for MVP. No leading zeros (except 0 itself — to avoid octal ambiguity). Digits required on both sides of the decimal point for floats.
    INT ::= '0' | [1-9][0-9]*
    FLOAT ::= INT '.' [0-9]+
    Note: .5 is NOT a valid EAML float — require leading digit. This avoids the .field vs .5 ambiguity.
    Critical
  • Define the negative number ambiguity resolution. Is -1.0 a unary minus applied to 1.0, or a single negative float token? Define which and document it.
    Recommendation: negative numbers are unary minus + positive literal, resolved by the parser. Never emit negative number tokens from the lexer — this would require context in the lexer, violating the separation of concerns.
    Critical
Operators & Punctuation
  • Enumerate every multi-character operator and verify maximal munch produces the correct token. Every multi-char operator must be defined so the lexer doesn't split it.
    EAML multi-char operators: -> (ARROW), == (EQEQ), != (NEQ), <= (LEQ), >= (GEQ), && (AND), || (OR), ?? (COALESCE), >> (PIPELINE, reserve even if Post-MVP). List every one. Test: does the lexer on -> emit ARROW, or MINUS + GT?
    Critical
  • Resolve the | pipe ambiguity in advance. | is used in literal union types AND potentially as a pipeline separator. A single | token is emitted by the lexer in both cases; the parser resolves meaning from context. Document this explicitly.
    Parser rule: in a TypeExpr position, | is a union separator. In an Expr position, | is bitwise OR (if supported) or a syntax error. Recommendation: do not support bitwise OR — | is type-union only, eliminating the ambiguity.
    Critical
  • Resolve the < / > angle bracket / comparison operator ambiguity. The lexer always emits LT and GT. The parser resolves type-param vs. comparison from context. Document the exact context rule.
    Rule: < is a type parameter opener ONLY when the parser is in a "type expression expected" state following a type name. In all other positions, < is the less-than comparison operator. No ambiguity exists IF type and value expressions are in mutually exclusive grammar positions — verify this for every occurrence.
    Critical
03📋 Declaration Grammar — model, schema, prompt, tool, agent, pipeline 0/0▼
model Declaration
  • Define the complete model declaration grammar. Name, API ID string, capability list. What is optional? What is required? Is a model without capabilities valid?
    ModelDecl ::= 'model' IDENT '=' 'Model' '(' STRING ',' 'caps' ':' '[' CapList ']' ')' ';'?
    Decision: caps is required — a model without declared capabilities cannot be checked against prompt requirements. Document this constraint.
    Critical
  • Define capability name grammar — closed enum vs. open identifiers. Closed: the grammar lists all valid capability names as keyword alternatives. Open: any identifier, validated in semantic analysis against a known set.
    Recommendation: open identifiers in grammar, validated against a registry in semantic analysis. CapList ::= IDENT (',' IDENT)*. This allows future capabilities without grammar changes.
    Critical
  • Define and reserve the model fallback chain syntax. Even if Post-MVP, define ?? as the fallback operator so existing code doesn't break when it's added.
    Reserve: ModelExpr ::= ModelRef ('??' ModelRef)*. Tag as Post-MVP in grammar comments.
    Medium
schema Declaration
  • Define the schema body field separator rules. Comma-separated, newline-separated, or either? Is a trailing separator allowed after the last field? Define this precisely.
    Recommendation: newline-or-comma, trailing separator allowed. FieldDef (',' | NEWLINE) FieldDef* (',' | NEWLINE)?. This matches user intuition and avoids "missing comma" errors that plague JSON-style formats.
    Critical
  • Define whether schemas can extend other schemas and reserve the keyword. Even if inheritance is Post-MVP, reserve extends as a keyword now to prevent it becoming a valid schema field name.
    High
  • Define field annotation syntax and reserve the @ sigil. @description("...") annotations on schema fields. Even if Post-MVP, reserve @ to prevent it becoming an operator.
    FieldDef ::= Annotation* IDENT ':' TypeExpr where Annotation ::= '@' IDENT ('(' ArgList ')')?. Mark annotations as Post-MVP in grammar but include the production.
    High
  • Define whether an empty schema is valid. schema Empty { } — is this a parse error or a semantic warning? What does it generate in the Python output?
    Medium
prompt Declaration
  • Define the complete prompt declaration grammar with optionality of each part marked explicitly. Parameters, defaults, requires clause, return type, body.
    PromptDecl ::= 'prompt' IDENT '(' ParamList? ')' RequiresClause? '->' TypeExpr PromptBody
    Required: name, (), ->, return type, body with at least user: field. Optional: params, requires clause, system field, temperature, max_tokens.
    Critical
  • Define prompt body field validity, required vs. optional fields, and ordering constraints.
    Required: user: TemplateStr. Optional: system: TemplateStr, temperature: FLOAT, max_tokens: INT. Ordering: document whether body fields must appear in a specific order or can be in any order (recommendation: any order — validated in semantic analysis).
    Critical
  • Define the requires clause grammar for single and multiple capabilities. Single: requires json_mode. Multiple: requires [json_mode, tools]. Pick one syntax or support both.
    Recommendation: RequiresClause ::= 'requires' (IDENT | '[' IDENT (',' IDENT)* ']') — single without brackets (ergonomic common case), multiple with brackets. Both emit the same AST node.
    High
  • Define whether empty parameter list () is required or whether parens can be omitted for zero-argument prompts.
    Recommendation: () always required. This makes every prompt declaration syntactically identical in structure, simplifying the parser and error messages.
    High
  • Define parameter default value grammar and which types can be defaults. lang: String = "English" — the default is a literal. Can it be an expression? A reference?
    Recommendation: literal defaults only for MVP. ParamDefault ::= '=' Literal. Expression defaults are Post-MVP — they require evaluation-order analysis.
    High
tool Declaration
  • Define the tool body grammar — native EAML body vs. Python bridge body as mutually exclusive alternatives.
    ToolBody ::= NativeBody | PythonBody
    NativeBody ::= '{' Statement* '}'
    PythonBody ::= 'python' '{' PYTHON_BLOCK '}'
    The parser must choose between these on the first token after { — if it sees the python keyword, enter python-block mode; otherwise parse statements.
    Critical
  • Define whether tools can have a void return type and the grammar for it. Side-effect-only tools. Option A: omit -> entirely. Option B: explicit -> Void.
    Recommendation: explicit -> Void. Omitting the arrow is ambiguous — looks like a forgotten return type. -> Void is intentional and self-documenting.
    High
  • Define the tool description field grammar. How does a developer specify the natural-language description the LLM sees for this tool?
    Recommendation: description: STRING as the first optional field in the tool body. Example: tool Foo(...) -> Bar { description: "Analyzes..." ... }. Visible in diffs, explicit in code.
    High
agent and pipeline
  • Define the agent declaration grammar with all valid fields.
    AgentDecl ::= 'agent' IDENT '{' AgentField* '}'
    AgentField ::= 'model' ':' ModelRef | 'tools' ':' '[' ToolRef (',' ToolRef)* ']' | 'system' ':' TemplateStr | 'max_turns' ':' INT | 'on_error' ':' ErrorPolicy
    High
  • Define and reserve the pipeline composition operator >>. Even if Post-MVP, reserve the token so two consecutive > comparisons don't accidentally parse as a pipeline.
    The lexer must emit >> as PIPELINE token when it sees two adjacent > characters, not as two GT tokens. This requires a maximal munch rule in the lexer.
    Medium
04🔷 Type Expression Grammar 0/0▼
Primitive Types
  • Enumerate all built-in primitive type names and decide: keywords or predeclared identifiers? Keywords prevent shadowing. Predeclared identifiers allow a schema named StringResult without conflict.
    Recommendation: predeclared type identifiers, not keywords. The symbol table initializes with String, Float, Int, Bool, Void pre-populated. A user can shadow them (with a warning). This is more ergonomic than reserving common English words.
    Critical
  • Define bounded type grammar precisely for each base type. What parameters does each built-in type accept? Are they positional or named?
    Float<min, max> — positional, both floats
    String<max: INT> — named, integer only
    Int<min: INT, max: INT> — both named, both optional
    Define the valid parameter names for each type explicitly. Semantic analysis enforces them.
    Critical
Composite Types & Precedence
  • Define literal union type grammar — minimum members, associativity, nesting.
    LiteralUnion ::= STRING_LIT ('|' STRING_LIT)+
    Minimum two members (one string is not a union). Left-associative by definition (it doesn't matter for unions, but document it). A single STRING_LIT in a type position is a string type constraint, not a union.
    Critical
  • Define optional type postfix ? and what it binds to. Tag[]? — optional array of Tag, or array of optional Tag? Define which and make the grammar unambiguous.
    Rule: postfix ? applies to the entire preceding TypeExpr at the current nesting level. Tag[]? = optional array of Tag. Tag?[] = array of optional Tag. Both are valid but mean different things — verify your grammar produces distinct AST nodes for each.
    Critical
  • Define a TypeExpr precedence hierarchy with explicit levels. This eliminates all type expression ambiguity by construction.
    Level 1 (base): NamedType | PrimitiveType | LiteralUnion | '(' TypeExpr ')'
    Level 2 (bounded): Level1 '<' BoundParams '>'
    Level 3 (array): Level2 '[]'
    Level 4 (optional): Level3 '?'
    Higher levels apply after lower levels. String<max:100>[]? parses as Optional(Array(Bounded(String, max:100))).
    Critical
  • Define whether type expressions can be parenthesized for grouping. ("a" | "b")[] — array of a literal union. Is this valid? If so, define GroupedType ::= '(' TypeExpr ')' as a Level 1 base type.
    High
  • Enumerate every grammar position where a TypeExpr can appear and verify each is reachable.
    Positions: (1) schema field type, (2) prompt return type after ->, (3) tool return type after ->, (4) param type annotation, (5) let binding annotation let x: T, (6) pipeline step constraint. Trace each to the TypeExpr production.
    Critical
  • Define array dimensionality limit for MVP. Single-dimension arrays only? If a user attempts Tag[][], emit a clear error rather than parsing it incorrectly.
    MVP: single-dimension only. The grammar production for ArrayType is Level2 '[]' — not recursive. A second [] immediately following should produce a parse error with message: "Multi-dimensional arrays are not supported in EAML v0.1."
    High
05✏️ Template String Grammar 0/0▼
Structure & Edge Cases
  • Define template strings as a distinct lexer mode producing a structured token sequence. Plain strings and template strings must be lexically distinguishable.
    TemplateStr ::= '"' TmplPart* '"'
    TmplPart ::= TmplText | TmplInterp
    TmplText ::= (~["{] | '{{' | '}}')+ — raw text, double-brace = literal brace
    TmplInterp ::= '{' IDENT '}' — single braces = interpolation
    The lexer emits: TMPL_START, (TMPL_TEXT | TMPL_INTERP)*, TMPL_END
    Critical
  • Define the escaped brace convention and document it visibly. How does a user include a literal { or } in a template string?
    Recommendation: {{ = literal {, }} = literal }. Matches Python f-strings and Jinja2 — familiar. Test: "JSON: {{ \"key\": \"value\" }}" should produce the literal string JSON: { "key": "value" }.
    Critical
  • Define what is valid inside an interpolation slot — identifier only or expressions.
    MVP: identifier only. TmplInterp ::= '{' IDENT '}'. Post-MVP: member access {{result.field}}. Keeping MVP identifier-only means interpolation resolution in semantic analysis is a simple name lookup — no expression evaluation needed.
    Critical
  • Define whether template strings can span multiple lines. Long system prompts routinely span multiple lines. If allowed, define how indentation whitespace is handled.
    Recommendation: allow multi-line. Content includes all characters including newlines. Optional: define a dedent behavior (trim common leading whitespace) as a compiler pass. Document whether indentation in multi-line templates is stripped or preserved literally.
    High
  • Define which string fields in declaration bodies are template strings vs. plain strings. Is every string a template string, or only specific field positions?
    Recommendation: positional rule. system: and user: fields in prompt bodies are always template strings. description: in tool bodies is a plain string (no interpolation needed). The grammar production for each field uses the appropriate terminal type.
    High
  • Define the behavior of an unclosed interpolation { IDENT (missing closing brace). Lexer error or parser error? What is the recovery strategy?
    Lexer error: emit TMPL_ERROR token with the offending position. The parser detects TMPL_ERROR and emits diagnostic "Unclosed interpolation in template string at line:col" and recovers to the next string boundary.
    High
06🐍 Python Bridge Grammar 0/0▼
Import Declarations
  • Define Python import declaration grammar precisely. File-level declarations. Module path as a STRING (not a dotted IDENT expression). Optional alias.
    PythonImport ::= 'import' 'python' STRING ('as' IDENT)? ';'?
    Example: import python "pandas" as pd
    The module path is always STRING — this sidesteps all dotted-name ambiguity. import python "sklearn.linear_model" as skl is unambiguous.
    Critical
  • Define whether Python imports can appear anywhere in the file or only at the top. Document as a semantic constraint (warning, not error) if mixing positions should be permitted.
    Medium
Python Block Lexer Specification
  • Define the Python block as a lexer-mode opaque capture region. The lexer enters a special mode on seeing python { and exits when the matching } is found at depth zero.
    PYTHON_BLOCK is emitted as a single token containing the raw Python source. Lexer algorithm: on python keyword + {, enter PYTHON_BLOCK mode. depth=1. For each { encountered: depth++. For each }: depth--. When depth==0: emit PYTHON_BLOCK(captured_text) and return to EAML mode.
    Critical
  • Define the brace-inside-Python-string-literal problem and your chosen resolution. This is the single hardest edge case in the Python bridge lexer.
    Three options — choose one and document:
    (A) Mini Python string scanner: when in PYTHON_BLOCK mode, track Python string literals (single/double/triple quoted) and don't count braces inside them. Correct but complex.
    (B) Delimiter syntax: require Python blocks to use a custom delimiter like python #{ ... }# that Python code cannot accidentally close. Simple and foolproof.
    (C) MVP limitation: document that Python code in python { } blocks must not contain unescaped literal } outside of braced constructs. Acceptable for MVP if documented clearly.
    Critical
  • Define whether Python blocks are syntax-validated by the EAML compiler. Opaque capture (no Python parsing) or optional py_compile post-pass?
    Recommendation: opaque for core compiler. Add an optional --check-python CLI flag that runs python -m py_compile on extracted blocks if Python is in PATH. Document this as "best effort" — the EAML compiler does not guarantee Python block correctness.
    High
  • Define the return type bridge contract between Python block and EAML type system.
    Grammar: the tool declaration's -> ReturnType creates the contract. Codegen: the emitted Python function wraps the python block's return value in ReturnType.model_validate(result). The grammar itself cannot enforce this — it is a runtime contract documented as a semantic rule: "The return value of a python { } block must be compatible with the tool's declared return type."
    High
07🔣 Expression Grammar & Operator Precedence 0/0▼
Operator Precedence Table — Define Before Writing Parser
  • Write out the complete operator precedence table, lowest to highest, with associativity for every operator. This table IS the expression grammar. Write it before coding anything.
    Recommended EAML table:
    1 || — left
    2 && — left
    3 == != — left, non-assoc
    4 < > <= >= — left, non-assoc
    5 + - — left
    6 * / — left
    7 Unary ! - — right
    8 . member access — left
    9 () call, [] index — left (highest)
    Document this verbatim in grammar.ebnf as a comment table.
    Critical
  • Define chained comparison operator behavior. Is 0.0 < x < 1.0 valid? Most languages parse this as (0.0 < x) < 1.0 (nonsensical). Define explicitly.
    Recommendation: non-associative comparisons. 0.0 < x < 1.0 is a parse error with hint: "Chained comparisons are not supported. Use: x > 0.0 && x < 1.0". Better error than silent wrong behavior.
    Critical
  • Define await as a prefix operator with explicit precedence.
    AwaitExpr ::= 'await' UnaryExpr — binds tighter than binary operators but looser than postfix. await a.call() + b is (await (a.call())) + b, not await ((a.call()) + b). Document this binding rule explicitly.
    High
  • Define member access chaining through call results. Claude.call(prompt).field — is postfix chaining valid after a call expression? Define how many levels can chain.
    PostfixExpr ::= PrimaryExpr Suffix* where Suffix ::= '.' IDENT | '(' ArgList ')' | '[' Expr ']'. Unlimited chaining, left-associative. This is standard and expected.
    High
Statement Grammar
  • Define the let binding grammar — is the type annotation required for MVP?
    Recommendation: required for MVP. LetDecl ::= 'let' IDENT ':' TypeExpr '=' Expr ';'?. Full type inference deferred to Post-MVP. Users must write let result: SentimentResult = ....
    Critical
  • Define semicolon policy — mandatory, optional, or automatic insertion.
    Recommendation: optional. Accept semicolons if present; do not require them. This is NOT ASI (no complex rules) — just tolerance. The parser determines statement end from the next token's FIRST set. Document: "Semicolons are optional statement terminators in EAML."
    High
  • Define if/else grammar and else-if chaining.
    IfStmt ::= 'if' Expr Block ('else' (IfStmt | Block))?
    The recursive else (IfStmt | Block) handles all chaining by natural recursion. Note: this is potentially ambiguous (the dangling-else problem). Resolve with: else always binds to the nearest preceding if. Document this rule.
    High
  • Define the dangling-else problem and its resolution for EAML.
    The dangling-else: if a if b X else Y — does else Y belong to the inner or outer if? Resolution: else always binds to the nearest enclosing if (greedy/eager binding). This is standard across C, Java, Rust. Document it as a grammar note.
    Critical
08⚠️ Explicit Ambiguity Resolution — Known Conflict Patterns 0/0▼
Structural Conflicts — Resolve Each One Explicitly
  • CONFLICT: { } as block vs. object literal. Does EAML have object literal syntax? If not, all { } are blocks and there is no conflict. Document this decision.
    Recommendation: no object literals. Schemas are the structured data construct. All { } are block delimiters. This eliminates the ambiguity entirely — a design win.
    Critical
  • CONFLICT: keyword model used as identifier in expression context. Claude.call() where Claude is a model-declared value, vs. starting a new declaration.
    Resolution: model keyword + IDENT token = ModelDecl. IDENT token (even if it names a model) in expression position = IdentExpr looked up in symbol table. The parser distinguishes by checking: am I at a declaration-start position?
    Critical
  • CONFLICT: -> in return type vs. potential function type syntax. Does EAML have function types like (String) -> Int? If not, -> only appears in declarations. Document whether function types exist.
    Recommendation: no function types in MVP. -> appears exclusively after parameter lists in declarations. Zero ambiguity in the grammar. Post-MVP: function types can be added as a TypeExpr production.
    Critical
  • CONFLICT: | in union type vs. bitwise OR. Define whether bitwise OR exists in EAML. If not, | is exclusively a union separator in type contexts.
    Recommendation: no bitwise OR in EAML. | is valid ONLY in TypeExpr positions as a union separator. In Expr positions, | is a syntax error with hint: "Did you mean '||' for logical OR?" Eliminates the ambiguity entirely.
    Critical
  • CONFLICT: </> angle brackets vs. comparison operators. Resolved by grammar position — define the rule precisely.
    Rule: < is a type parameter delimiter ONLY when the parser is evaluating a TypeExpr and has just parsed a TypeName. In all Expr contexts, < is always less-than. Verify: TypeExpr and Expr contexts are ALWAYS mutually exclusive in the grammar. If any rule allows both in the same position, redesign it.
    Critical
  • CONFLICT: built-in type name vs. user schema name (shadowing). schema String { ... } shadows the built-in. Document the resolution rule.
    Resolution: user declarations shadow built-in types in the symbol table. Emit a warning: "Declaration 'String' shadows built-in type. Use a different name to avoid confusion." The declaration is valid — the warning is advisory.
    High
  • CONFLICT: empty requires list vs. absent requires clause. requires [] vs. no requires clause. Are these semantically equivalent? Define the grammar distinction.
    Recommendation: make them equivalent. requires [] and absent requires both mean "no capability requirements." The AST uses a requires: Vec<Symbol> field that is empty in both cases. Document this so the emitted code handles both correctly.
    High
Left-Recursion & LL(k) Validity
  • Verify the grammar has zero left-recursive productions. Left recursion causes infinite loops in recursive descent. Check every production.
    Check: for every rule A ::= ..., can A derive a string starting with A? If yes: left recursion. Fix: rewrite as right recursion or as an iterative rule with { ... } (repetition). The expression grammar is the most common source — use Pratt parsing to avoid left recursion in expression rules.
    Critical
  • Compute FIRST and FOLLOW sets for every non-terminal production with multiple alternatives.
    A grammar is LL(1) if the FIRST sets of all alternatives for every rule are disjoint. Compute these for: TypeExpr alternatives, Declaration alternatives, PromptBody field alternatives, Expr alternatives. Use a grammar analysis tool or compute manually and verify on paper.
    Critical
  • Document every rule that requires LL(2) lookahead and verify no rule requires LL(3+).
    Expected LL(2) points: BoundedType vs. TypeName followed by comparison operator. Every LL(2) point must be documented with: the two alternatives, the distinguishing token sequence, and the rule for the parser to follow.
    Critical
09🚑 Error Recovery Grammar 0/0▼
Recovery Strategy
  • Define primary synchronization tokens for declaration-level error recovery. When the parser encounters an unexpected token inside a declaration, these tokens are the "safe points" to skip to.
    Primary sync set: model, schema, prompt, tool, agent, pipeline, import, EOF. Strategy: on declaration-level error, consume tokens until one of these is seen, then resume parsing the next declaration. At most one declaration is lost per error.
    Critical
  • Define secondary synchronization for schema field-level recovery. When a field definition is malformed, recover to the next field rather than the next declaration.
    Field-level sync: on field parse error, skip to the next , (field separator) or } (schema close). This allows other fields in the same schema to parse correctly.
    High
  • Define whether ErrorNode appears in the AST or errors are silent.
    Recommendation: include ErrorNode { span: Span, message: String } in the AST. Downstream compiler phases check for ErrorNode and suppress cascading errors on subtrees that contain one. Without ErrorNode, malformed subtrees produce phantom errors in semantic analysis.
    High
  • Define a maximum error count threshold before the compiler aborts.
    Default: 20 errors. Configurable via --max-errors N. After N errors, emit "Too many errors. Fix the above and recompile." — continuing to generate hundreds of cascading errors degrades UX significantly.
    Medium
  • Define panic mode vs. phrase-level recovery strategy. Panic mode: discard tokens until sync token. Phrase-level: insert or delete a single token to fix the error. Document which strategy EAML uses and where each applies.
    Recommendation: panic mode at declaration level (simple, predictable), phrase-level for common mistakes at expression level (e.g., missing ) at end of parameter list). The latter gives better error messages for the most common typos.
    Medium
