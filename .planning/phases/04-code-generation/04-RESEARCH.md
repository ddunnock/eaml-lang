# Phase 4: Code Generation - Research

**Researched:** 2026-03-16
**Domain:** Rust-based Python code generation from AST (compiler backend)
**Confidence:** HIGH

## Summary

Phase 4 implements the `eaml-codegen` crate, which walks a semantically-validated AST and emits Python 3.11+ / Pydantic v2 code. The crate exists as a stub (`lib.rs` with only a doc comment) with dependencies already wired: `eaml-errors`, `eaml-lexer`, `eaml-parser`, `eaml-semantic`, and `insta` for snapshot testing.

The input contract is well-defined: `ParseOutput` (AST + interner) and `AnalysisOutput` (symbol table + type annotations) from upstream phases. The output is a `String` containing a complete Python module. The type mapping table in `spec/TYPESYSTEM.md` section 10.3 provides an authoritative, exhaustive mapping from every EAML type expression to its Python annotation and Pydantic Field() call. The Python bridge spec (`spec/PYTHON_BRIDGE.md` section 5) provides the exact generated code structure for tools including bridge functions, wrappers, and metadata registration.

**Primary recommendation:** Build a hand-written `CodeWriter` struct with indent/dedent tracking, then implement one emitter method per declaration type (schema, model, prompt, tool, agent, let). Use `insta` snapshot tests against the example `.eaml` files as the primary verification mechanism.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- One `.py` file per `.eaml` file (e.g., `sentiment.eaml` -> `sentiment.py`)
- Declarations emitted in dependency order: imports, let bindings, schemas, models, prompts/tools, agents
- Section header comments between groups: `# --- Schemas ---`, `# --- Models ---`, etc.
- File header comment: `# Generated by eamlc from <filename>.eaml`
- Schema names stay PascalCase (Python class convention)
- Prompt and tool names convert to snake_case functions (`AnalyzeSentiment` -> `analyze_sentiment`)
- Model names become UPPER_SNAKE_CASE config dicts (`Sonnet` -> `SONNET_CONFIG`)
- Agent names stay PascalCase (Python class convention)
- Tool parameters keep their EAML names as-is (no conversion)
- Bridge code extracted from `code_span`, dedented to normalize indentation, pasted as function body
- Tool functions are sync `def` (not `async def`)
- Tool descriptions (if present) emitted as Python triple-quote docstrings
- Models generate plain config dicts with `provider`, `model_id`, and `capabilities` keys
- Prompts generate `async def` functions that construct message lists and call `eaml_runtime.execute_prompt()`
- `execute_prompt()` receives: model config dict, messages list, return_type (Pydantic class), and options (temperature, max_tokens, max_retries)
- Agents generate classes extending `eaml_runtime.Agent` base class with attributes: `model`, `tools`, `system_prompt`, `max_turns`, `on_error`
- Runtime handles provider dispatch, Pydantic validation, and retry logic -- codegen does not emit retry loops

### Pattern 1: CodeWriter with Indent Tracking
**What:** A struct that manages a `String` buffer with indentation state.
**When to use:** All Python code emission.
**Example:**
```rust
pub struct CodeWriter {
    buf: String,
    indent: usize,       // Current indentation level (number of 4-space indents)
    at_line_start: bool,  // Whether next write should prepend indentation
}

impl CodeWriter {
    pub fn new() -> Self { ... }
    pub fn indent(&mut self) { self.indent += 1; }
    pub fn dedent(&mut self) { self.indent -= 1; }
    pub fn write(&mut self, s: &str) { ... }  // Write text, prepend indent if at line start
    pub fn writeln(&mut self, s: &str) { ... } // Write text + newline
    pub fn blank_line(&mut self) { ... }       // Empty line (no indentation)
    pub fn finish(self) -> String { self.buf }
}
```

### Pattern 2: Import Collector
**What:** Track needed imports during AST traversal, emit deduplicated at the top.
**When to use:** GEN-10 import deduplication.
**Example:**
```rust
struct ImportCollector {
    pydantic: HashSet<&'static str>,  // "BaseModel", "Field"
    typing: HashSet<&'static str>,    // "Optional", "List", "Literal"
    eaml_runtime: HashSet<&'static str>, // "execute_prompt", "Agent", "ToolMetadata"
    python_imports: Vec<String>,       // from `import python "..."` declarations
}
```

### Pattern 3: Two-Pass Generation
**What:** First pass collects imports and reorders declarations; second pass emits code.
**When to use:** Ensuring imports are complete and declarations are in dependency order.
**Details:**
1. **Pass 1:** Walk all declarations, collect which imports are needed (e.g., if any schema has bounded fields, need `Field` from pydantic; if any has Optional, need `Optional` from typing). Reorder declarations into: imports -> lets -> schemas -> models -> prompts/tools -> agents.
2. **Pass 2:** Emit file header, imports, then each declaration group with section comments.

### Pattern 4: Type Emission via ResolvedType
**What:** Use the `ResolvedType` enum from semantic analysis to drive Python type annotation generation.
**When to use:** Schema fields, function parameters, return types.
**Key mapping:**
```rust
fn emit_type_annotation(resolved: &ResolvedType, ast: &Ast, interner: &Interner) -> String {
    match resolved {
        ResolvedType::Primitive(name) => match name.as_str() {
            "string" => "str".to_string(),
            "int" => "int".to_string(),
            "float" => "float".to_string(),
            "bool" => "bool".to_string(),
            "null" => "None".to_string(),
            _ => unreachable!(),
        },
        ResolvedType::Schema(id) => interner.resolve(&ast[*id].name).to_string(),
        ResolvedType::Array(inner) => format!("List[{}]", emit_type_annotation(inner, ...)),
        ResolvedType::Optional(inner) => format!("Optional[{}]", emit_type_annotation(inner, ...)),
        ResolvedType::LiteralUnion(members) => {
            let quoted: Vec<_> = members.iter().map(|m| format!("\"{}\"", m)).collect();
            format!("Literal[{}]", quoted.join(", "))
        },
        ResolvedType::Error => "Any".to_string(), // Fallback, should not occur after clean analysis
    }
}
```

### Anti-Patterns to Avoid
- **String concatenation without indent tracking:** Leads to broken Python indentation. Always use CodeWriter.
- **Emitting imports inline with declarations:** Causes duplicate imports. Always use ImportCollector + top-of-file emission.
- **Hardcoding type mappings in multiple places:** The type mapping logic should be in one function (`types.rs`), not scattered across emitters.
- **Not using `ResolvedType` from semantic analysis:** The type checker already resolved all types. Re-resolving from raw AST `TypeExpr` nodes is duplicated work and may miss edge cases.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Type annotation generation | Ad-hoc string formatting per call site | Single `emit_type_annotation()` function using `ResolvedType` | 33+ type forms in mapping table; centralized = testable |
| PascalCase to snake_case | Regex or ad-hoc char iteration | Dedicated `to_snake_case()` with proper handling of consecutive capitals | Edge cases: `HTTPClient` -> `http_client`, `XMLParser` -> `xml_parser` |
| Python bridge dedent | Manual whitespace trimming | `textwrap::dedent` equivalent: find minimum indent, strip that prefix from all lines | Python is indent-sensitive; getting this wrong breaks the output |
| Bounded type Field() generation | Inline if/else chains | Lookup table mapping bound param names to Pydantic Field kwargs | `min`/`max` -> `ge`/`le` for numeric, `minLen`/`maxLen` -> `min_length`/`max_length` for string |

**Key insight:** The type mapping table in TYPESYSTEM.md 10.3 is exhaustive and authoritative. Codegen must implement it exactly, not approximate it.

## Common Pitfalls

### Pitfall 1: Optional Fields Without `= None` Default
**What goes wrong:** Pydantic v2 treats `Optional[T]` without a default as a required field that accepts None. EAML semantics say `T?` means the field is truly optional.
**Why it happens:** Confusion between "can be None" and "can be omitted."
**How to avoid:** Per TYPESYSTEM.md rule TS-GEN-01, any field whose type is `Optional[...]` MUST also have `= None` default value.
**Warning signs:** mypy passes but Pydantic validation fails on LLM responses missing optional fields.

### Pitfall 2: Python Bridge Indentation
**What goes wrong:** Bridge code extracted from source text has EAML-level indentation baked in. If pasted verbatim, the Python function body has extra leading whitespace.
**Why it happens:** The `code_span` in `ToolBody::PythonBridge` captures the raw source including EAML indentation.
**How to avoid:** Implement a dedent algorithm: split bridge content into lines, find minimum non-empty-line indent, strip that prefix from all lines. Then re-indent to the function body level (2 indents = 8 spaces).
**Warning signs:** Python `IndentationError` when importing generated module.

### Pitfall 3: Template String to F-String Escaping
**What goes wrong:** EAML template strings use `{expr}` for interpolation, same as Python f-strings. But literal braces in EAML use `{{`/`}}`, which also matches Python f-string escaping. Edge case: if the template text itself contains `{` or `}` that are NOT interpolations, they need to be escaped as `{{`/`}}` in the f-string output.
**Why it happens:** EAML and Python use the same brace interpolation syntax.
**How to avoid:** When emitting `TemplatePart::Text`, escape any `{` or `}` in the text as `{{`/`}}`. When emitting `TemplatePart::Interpolation`, emit the expression inside `{...}` unescaped.
**Warning signs:** Python `ValueError` or `KeyError` from f-string evaluation.

### Pitfall 4: Import Deduplication with Conditional Imports
**What goes wrong:** Emitting `from typing import Optional, List, Literal` when the file only uses `str` and `int` -- unused imports fail linting.
**Why it happens:** Blanket import emission without tracking which types are actually used.
**How to avoid:** ImportCollector tracks which specific imports are needed. Only emit `Optional` if at least one `T?` type appears. Only emit `Literal` if at least one literal union appears. Only emit `Field` if at least one bounded type appears.
**Warning signs:** mypy warnings about unused imports (if strict mode), or just messy output.

### Pitfall 5: Declaration Ordering for Forward References
**What goes wrong:** A schema `Person` references `Address`, but `Address` is defined after `Person` in source. If emitted in source order, Python raises `NameError`.
**Why it happens:** Python classes are evaluated top-down; forward references aren't available.
**How to avoid:** Topological sort schemas by dependency. If cycles exist (recursive schemas, which get SEM070 warning), emit `model_rebuild()` calls after all schema definitions, or use string-based forward references (`"Address"` instead of `Address`). For v0.1, topological sort is sufficient since recursive schemas are warned but allowed.
**Warning signs:** `NameError: name 'Address' is not defined` when importing generated module.

### Pitfall 6: Bounded Type Parameter Extraction from Source Text
**What goes wrong:** Bounded type values are stored as `Span` (byte ranges into source), not as parsed numbers. Need to extract the actual numeric value from the source string.
**Why it happens:** The AST stores `BoundParam { value_span: Span }`, not a parsed float/int.
**How to avoid:** Use `source[param.value_span.clone()]` to extract the string, then format it appropriately for Python `Field()` kwargs. The type checker already validated these are valid numbers.
**Warning signs:** Emitting `Field(ge=)` with empty values, or wrong values.

async def analyze_sentiment(text: str) -> SentimentResult:

        {"role": "system", "content": "You are a sentiment analysis expert..."},

        max_retries=3,

## Open Questions

1. **Prompt model binding at codegen time**
   - What we know: Prompts have `requires` clauses but don't directly name a model. At runtime, the caller must pass a model config.
   - What's unclear: Should the generated prompt function accept a `model` parameter, or should it reference a default model from the file? The CONTEXT.md says `execute_prompt()` receives a model config dict, suggesting the prompt function takes it as a parameter.
   - Recommendation: Generate prompt functions with a `model` parameter (the config dict). The caller passes which model to use.

2. **Error AST node handling**
   - What we know: `DeclId::Error(Span)` and `Expr::Error(Span)` exist for recovery.
   - What's unclear: Whether to skip silently or emit a comment.
   - Recommendation: Skip `Error` nodes silently. Codegen should only run after successful semantic analysis (no fatal errors). If errors exist, the CLI should abort before codegen.

3. **Agent `on_error` field representation**
   - What we know: `ErrorPolicy::Fail` and `ErrorPolicy::RetryThenFail { retries_span }` are the AST variants.
   - What's unclear: Exact Python attribute format for the retry count.
   - Recommendation: Emit `on_error = "fail"` or `on_error = "retry_then_fail"` with a separate `on_error_retries = N` attribute for the retry variant. Extract N from `retries_span` using source text.
