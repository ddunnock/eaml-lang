# Phase 4: Code Generation - Context

**Gathered:** 2026-03-16

<domain>
## Phase Boundary

The compiler emits valid, runnable Python 3.11+ / Pydantic v2 code from a semantically-validated AST. Covers all five declaration types (model, schema, prompt, tool, agent) plus let bindings. The Python runtime implementation is Phase 5 -- codegen only needs to emit the correct `eaml_runtime` API calls.

</domain>

<decisions>
## Implementation Decisions

### Output file structure
- One `.py` file per `.eaml` file (e.g., `sentiment.eaml` -> `sentiment.py`)
- Declarations emitted in dependency order: imports, let bindings, schemas, models, prompts/tools, agents
- Section header comments between groups: `# --- Schemas ---`, `# --- Models ---`, etc.
- File header comment: `# Generated by eamlc from <filename>.eaml`

### Python naming conventions
- Schema names stay PascalCase (Python class convention)
- Prompt and tool names convert to snake_case functions (`AnalyzeSentiment` -> `analyze_sentiment`)
- Model names become UPPER_SNAKE_CASE config dicts (`Sonnet` -> `SONNET_CONFIG`)
- Agent names stay PascalCase (Python class convention)
- Tool parameters keep their EAML names as-is (no conversion)

### Python bridge emission
- Bridge code extracted from `code_span`, dedented to normalize indentation, pasted as function body
- Tool functions are sync `def` (not `async def`)
- Tool descriptions (if present) emitted as Python triple-quote docstrings

### Model representation
- Models generate plain config dicts with `provider`, `model_id`, and `capabilities` keys
- Not dataclasses or runtime classes -- simple dicts consumed by `eaml_runtime.execute_prompt()`

### Prompt execution pattern
- Prompts generate `async def` functions that construct message lists and call `eaml_runtime.execute_prompt()`
- `execute_prompt()` receives: model config dict, messages list, return_type (Pydantic class), and options (temperature, max_tokens, max_retries)
- Runtime handles provider dispatch, Pydantic validation, and retry logic -- codegen does not emit retry loops

### Agent generation
- Agents generate classes extending `eaml_runtime.Agent` base class
- Class attributes: `model`, `tools` (list of tool function references), `system_prompt`, `max_turns`, `on_error`
- Runtime base class handles the orchestration loop

### Claude's Discretion
- Exact CodeWriter API design (method names, buffer strategy)
- Import deduplication algorithm
- PascalCase-to-snake_case conversion edge cases
- How to handle Error AST nodes during codegen (skip or emit comment)
- Exact dedent algorithm for Python bridge blocks

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `eaml-codegen` crate exists as a stub (`lib.rs` with doc comment only) -- clean slate for implementation
- `Ast` typed arenas with `Index` impls -- codegen can index directly into arenas via ID newtypes
- `SymbolTable` with `get()`, `is_primitive()`, `is_known_type()` -- useful for type resolution during codegen
- `TypeAnnotations` side table from semantic analysis -- provides validated type information
- `Interner` (lasso) for resolving `Spur` back to string names via `resolve()`
- `insta` crate already in workspace deps for snapshot testing

### Established Patterns
- Typed ID newtypes (`ExprId`, `TypeExprId`, `ModelDeclId`, etc.) for arena indexing
- `Program.declarations` gives source-order traversal; codegen reorders to dependency order
- `TemplateString.parts` with `TemplatePart::Text` and `TemplatePart::Interpolation` for f-string generation
- `BoundParam` with `value_span` for extracting literal values from source text

### Integration Points
- Input: `ParseOutput` (from eaml-parser) + `AnalysisOutput` (from eaml-semantic) + source text
- Output: `String` containing complete Python module
- Will be consumed by CLI (Phase 6) which writes the string to a `.py` file
- Generated code imports from `eaml_runtime` -- defines the API contract Phase 5 must implement

</code_context>
